# 2차 프로젝트 요구사항 정리

## 프로젝트 개요

현대퓨처넷 메시징 시스템을 주제로 한 2차 팀 프로젝트입니다. 4명이 2주 동안 진행하는 것을 전제로 하며, 실제 구현 단계에서는 Spring Boot 기반 웹 애플리케이션으로 개발합니다.

현재 저장소는 백엔드 구현 전 기능 범위와 화면 흐름을 검토하기 위한 프론트엔드 데모입니다. 향후 백엔드 프로젝트로 옮길 때 이 문서를 요구사항 기준으로 사용합니다.

## 팀/기간

- 인원: 4명
- 기간: 2주
- 목표: 메시징 운영자가 회원, 템플릿, 발송, 기록, 통계를 관리할 수 있는 사내 운영 시스템 구현
- 결과물: 웹 애플리케이션, DB 설계, 주요 기능 시연, 발표 자료

## 기술 스택

- Java

- Thymeleaf

## 핵심 사용자

### 운영 관리자

- 시스템 전체 기능 접근
- 회원 정보 확인 및 수정
- 메시지 템플릿 관리
- 발송 요청 생성
- 전송 기록 및 통계 확인

### 마케팅 담당자

- 캠페인 대상 선정
- 템플릿 선택 및 문구 작성
- AI 문구 추천/검사 결과 확인
- 발송 성과 분석

### 고위 의사결정자

- 발송량, 성공률, 채널 효율, 회원 반응률 확인
- 마케팅 전략 판단을 위한 통계 확인

## 주요 기능 요구사항

### 1. 로그인/보안

필수 기능:

- 로그인
- 로그아웃
- Spring Security 적용
- 비밀번호 암호화
- 권한별 접근 제어
- 개인정보 마스킹

검토 사항:

- 회원가입 화면은 만들지 않음
- 운영자 계정은 DB에 직접 등록하거나 관리자 등록 기능으로 처리
- 전화번호, 이름 등 개인정보는 목록에서 마스킹 표시
- 상세 조회 시에도 권한에 따라 노출 범위 제한

### 2. 대시보드

필수 기능:

- 오늘 발송 건수
- 발송 성공률
- 활성 회원 수
- 월 누적 발송 수
- 최근 전송 기록
- 채널별 발송 비중
- 일별 발송 추이

추가 권장:

- 실패 건수 요약
- 예약 발송 대기 건수
- AI 검사 주의/위험 건수
- 이번 주 성과 인사이트
- 빠른 작업 버튼

### 3. 메시지 발송

가장 중요한 핵심 플로우입니다.

단계:

1. 수신자 선택
2. 메시지 작성
3. 채널 선택
4. AI 검사 및 최종 발송

수신자 선택 요구사항:

- 태그 기반 대상 선택
- 태그 검색
- 유사 태그 추천
- 회원 개별 검색
- 회원 개별 포함
- 회원 개별 제외
- 수동 수신자 추가
- 회원은 여러 개의 태그를 가질 수 있음
- 전체 회원 수가 수천 명 이상이어도 사용할 수 있는 UI 필요

메시지 작성 요구사항:

- 템플릿 검색
- 템플릿 선택
- 템플릿 미리보기
- 변수 치환 미리보기
- 직접 메시지 작성
- 템플릿 수가 수천~수만 개여도 검색/필터로 찾을 수 있어야 함

채널 선택 요구사항:

- SMS
- LMS
- RCS
- 카카오 알림톡
- 카카오 친구톡

AI 추천 요구사항:

- AI가 대상, 템플릿, 채널, 문구를 한 번에 추천
- 담당자는 추천안을 그대로 쓰거나 수정 가능
- AI 추천 결과는 사람이 검토한 뒤 발송

AI 검사 요구사항:

- 발송 전 AI 검사를 반드시 거치는 흐름
- 여러 개의 작은 LLM을 비동기로 실행하는 구조를 가정
- 검사 항목:
  - 오타/맞춤법
  - 광고 표기
  - 민감 표현
  - 사회적 이슈 문구
  - 개인정보 노출
  - 마스킹 필요 여부
  - 채널별 글자 수 적합성
  - 발송 피로도
- 간단 요약 리포트와 상세 리포트 제공
- 주의/위험 결과가 있으면 담당자가 수정 후 재검사

발송 요구사항:

- 즉시 발송
- 예약 발송
- 최종 확인
- 발송 요청 생성
- 실제 프로젝트에서는 발송 API 연동 대신 mock 처리 가능

### 4. 템플릿 관리

필수 기능:

- 템플릿 목록
- 템플릿 검색
- 템플릿 추가
- 템플릿 수정
- 템플릿 삭제
- 템플릿 상세 팝업
- 페이지네이션

필드 예시:

- 템플릿명
- 채널
- 카테고리
- 태그
- 메시지 본문
- 변수 목록
- 사용 횟수
- 수정일
- 사용 여부

추가 권장:

- AI 문구 추천
- AI 문구 검사
- 템플릿 복사
- 최근 사용 템플릿
- 고성과 템플릿 표시

### 5. 전송 기록

필수 기능:

- 전송 기록 목록
- 검색/필터
- 행 클릭 상세 팝업
- 페이지네이션

필드 예시:

- 발송 ID
- 캠페인명
- 템플릿명
- 채널
- 대상 조건
- 발송 수
- 성공 수
- 실패 수
- 성공률
- 발송 일시
- 상태

상세 정보:

- 수신자 조건
- 사용 템플릿
- 실제 메시지
- AI 검사 결과
- 실패 사유
- 채널별 결과

### 6. 회원 관리

필수 기능:

- 회원 목록
- 회원 검색
- 회원 상세 팝업
- 회원 정보 수정
- 수신 동의 여부 확인
- 태그 확인
- 페이지네이션

회원 정보:

- 회원 ID
- 이름
- 전화번호
- 회원 유형
- 보유 태그
- SMS 동의
- LMS 동의
- RCS 동의
- 카카오 동의
- 가입일
- 최근 발송일

변경 기능:

- 개별 회원 수정
- 선택 회원 부분 변경
- 전체 조건 대상 일괄 변경
- 태그 추가/삭제
- 수신 동의 상태 변경

파일 업로드:

- Excel 업로드
- CSV 업로드
- JSON 업로드
- 업로드 전 검증
- 업로드 결과 리포트
- 실패 행 확인

주의 사항:

- 담당자가 실수하기 쉬운 기능이므로 확인 단계 필요
- 전체 변경과 부분 변경은 명확히 구분
- 대량 변경 시 예상 변경 건수 표시

### 7. 통계

통계는 단일 그래프가 아니라 여러 페이지로 구성합니다.

필수 페이지:

- 통계 개요
- 채널 통계
- 회원/태그 통계
- 캠페인 성과 통계

통계 개요:

- 전체 발송 수
- 성공률
- 실패율
- 클릭률 또는 반응률
- 채널별 비중
- 기간별 발송 추이

채널 통계:

- SMS/LMS/RCS/카카오별 발송량
- 채널별 성공률
- 채널별 실패율
- 채널별 반응률
- 채널별 비용 효율

회원/태그 통계:

- 태그별 반응률
- 회원 유형별 발송 성과
- 동의 채널별 회원 분포
- 휴면/신규/VIP 등 세그먼트별 성과

캠페인 성과 통계:

- 캠페인별 발송량
- 캠페인별 성공률
- 캠페인별 반응률
- 템플릿별 성과
- 고성과/저성과 문구 비교

의사결정에 필요한 정보:

- 어떤 채널이 가장 효율적인가
- 어떤 태그/회원군이 반응이 좋은가
- 어떤 템플릿 문구가 효과적인가
- 실패가 많은 채널이나 시간대는 어디인가
- 다음 캠페인에서 제외하거나 집중할 대상은 누구인가

## 화면 UX 요구사항

공통:

- 불필요한 설명/장식 제거
- 운영자가 실수하지 않도록 정보 구조 명확화
- 표 행 hover 시 색상 변경
- 클릭 가능한 행은 pointer cursor
- 대량 데이터를 고려한 검색/필터/페이지네이션
- 삭제/발송/대량 변경은 확인 단계 필요

메시지 발송:

- 단계형 플로우 유지
- AI 추천 유지
- 회원 개별 포함/제외 유지
- 템플릿은 단순 드롭다운 금지
- 태그는 전체 나열 금지
- AI 검사는 중요 기능으로 상세화

디자인:

- 깔끔하고 현대적인 운영 시스템
- 밝은 배경
- Pretendard 계열 폰트
- 과도한 굵기 금지
- 그라데이션 지양
- 장식적 이미지/문구 지양
- 주요 액션과 상태만 색상 강조

## 발표 시 강조 포인트

- 단순 문자 발송이 아니라 마케팅 운영 시스템
- 대량 회원/태그/템플릿 운영을 고려한 UI
- AI 문구 추천과 AI 리스크 검사를 발송 플로우에 통합
- 담당자 실수를 줄이는 단계형 발송 구조
- 전송 결과를 통계로 다시 마케팅 전략에 반영하는 구조

## 현재 프론트 데모와의 연결

현재 데모에서 확인할 수 있는 요구사항:

- 로그인 화면
- 대시보드
- 단계형 메시지 발송
- AI 자동 추천
- AI 비동기 검사
- 템플릿 관리
- 전송 기록 상세
- 회원 관리 및 파일 업로드 UI
- 4개 통계 화면

실제 구현 시 이 데모를 그대로 복제하기보다, 핵심 플로우와 정보 구조를 Spring Boot/Thymeleaf 화면으로 옮기는 것을 목표로 합니다.
